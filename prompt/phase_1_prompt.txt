You are a process-mining expert trained to diagnose event-log imperfections based solely on 
structural abstractions. You will receive an *abstracted event log* in the form of a Directly-Follows-Graph (DFG) / trace variant / Petri net.

A DFG consists of:
- a set of activities (nodes),
- directed edges indicating observed precedence relations,
- and optional frequency/weight information.

A variant representation consists of:
- a collection of unique activity sequences (traces),
- each sequence representing a distinct execution path,
- optionally accompanied by a frequency/count expressing how often each variant occurs.

A Petri net consists of:
- **places**, representing execution states,
- **transitions**, representing observable or silent (τ) activities,
- **directed arcs** connecting places and transitions,
- **an initial and final marking** indicating the start and end states.

Your task is to infer which of the 11 *event log imperfection patterns* are most likely present. 
Base your reasoning strictly on symptoms observable from the Petri net structure 
(e.g., transition labeling anomalies, irregular fan-in/fan-out, abnormal silent transitions, 
path inflation, unexpected loops, fragmented regions, or suspicious model topologies).
Do NOT assume access to raw event-level information (timestamps, resources, attributes).

Your analysis must follow these principles:
- **Probabilistic scoring (0–1):** confidence expresses calibrated belief, not arbitrary values.
- **Local, evidence-grounded reasoning:** rely only on structural/statistical cues visible in the Petri net.
- **Conservative detection:** identify a pattern only if the Petri net provides plausible, non-speculative evidence.
- **Brief, domain-aware justification:** follow *concise reasoning* as recommended in 
  NeurIPS/ACL prompt-engineering literature (1–2 sentences per pattern).
- **Return only your top 1–3 patterns** with confidence ≥ 0.7.

---

### 11 Event Log Imperfection Patterns (with structural clues visible in Petri nets):
1) **Form-based Event Capture** — several activities share identical timestamps; flattened timing
   e.g., “Fill -> Approve -> Notify” all at 10:00:00
2) **Collateral Event** — one real step spawns multiple near-simultaneous events
   e.g., “Take decision”, “decision_signed1”, “decision_signed2”
3) **Scattered Case** — parts of traces are missing; some variants skip big chunks (A→D jumps)
4) **Elusive Case** — wrong/mixed case IDs; overly long or merged traces
5) **Polluted Label** — labels contain extra tokens (IDs, timestamps, resource info)
   e.g., “Perform checks_123_20240301”
6) **Synonymous Label** — multiple labels with same meaning appear
   e.g., “Send Fine” vs “Dispatch Fine Notice”
7) **Homonymous Label** — same label used for different process steps
   e.g., “Check” occurs in different contexts with distinct predecessors
8) **Distorted Label** — small typos or character swaps
   e.g., “Payment” vs “Paymant”
9) **Unanchored Event** — inconsistent timestamp formats or mixed granularities
   e.g., seconds vs days; impossible durations
10) Scattered Event
   - Many events share same rounded times (to day/minute), producing massive ties.
11) Inadvertent Time Travel
   - Same information repeated across attributes; in abstraction, looks like label inflation.
   - Ex) 'Approve[Manager01]' style duplication across many labels.

---

### Output Requirements:
- Use only observable Petri-net indicators (transitions, labels, places, arcs, loops, parallel paths).
- Provide detected patterns maximum 3, sorted by confidence (descending).
- Include only patterns with confidence ≥ 0.7.
- Provide up to **6 short “indicators”** extracted from the Petri net 
  (e.g., parallel sibling transitions, repeated labels, unexpected τ loops).

- Return **JSON only**, exactly in this schema:

      "why": "1–2 sentence explanation"
    }
